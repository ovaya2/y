## Ausbilder-Hinweise — Umsetzung

### ✓ SQL-Normalisierung (1NF→3NF)

Schema korrekt normalisiert. FKs bewusst gesetzt. Index-Lücke eigenständig erkannt — geplant.

**Details:**  
Lookup-Tabellen (`d_anr`, `d_sta`, `d_ybart` etc.) trennen Stamm- von Bewegungsdaten sauber.

CASCADE nur wo fachlich sinnvoll: `f_soll → m_gl` (Mitglied löscht Gebühren mit), aber NO ACTION auf `f_ist` (Zahlungen bleiben für Kassenprüfung).

Eigenständig erkannt: `WHERE sta_id <> 6` und `WHERE erledigt = 0` laufen ohne Index. Geplant: `CREATE INDEX IX_m_gl_sta_id` und `IX_f_soll_erledigt`.

---

### ✓ Typ-Mapping (SQL → C#)

Nullable-Handling zentral via `FromDbNull<T>()`. Fehler behoben: `ZahlungsartId` aus Member entfernt.

**Details:**  
`SqlExtensions.FromDbNull<T>()` und `ToDbNull()` übersetzen zwischen C# `null` und SQL `DBNull.Value` — zentral, kein duplizierter Code in den Repositories.

Enum-Cast: `(MemberStatus)r["sta_id"]` — Werte entsprechen exakt den DB-IDs in `d_sta`.

Fehler v3.4: `ZahlungsartId` gehört zu `f_ist`, nicht zu `m_gl` — Property aus `Member.cs` entfernt.

---

### ✓ Model aus mehreren Tabellen

LEFT JOINs in einem DB-Roundtrip statt N+1-Queries.

**Details:**
- **Fees:** `f_soll LEFT JOIN f_ist` → `PaidAmount = COALESCE(SUM(betrag), 0)` — ein Query für alle Forderungen inkl. Zahlungsstand.
- **User:** `m_u LEFT JOIN m_gl` → `MemberName` direkt im Select.
- **RBAC-Login:** `m_u → x_ur → d_role → x_rp → d_perm` — kompletter Objektgraph (User + Rollen + Permissions) in einem einzigen Query, im C#-Code per Dictionary aggregiert.

---

### ✓ DataProvider ↔ Model

ViewModels kennen nur Interfaces. Verbindungslogik zentral in `SqlRepositoryBase`.

**Details:**  
Repository-Pattern: `IMemberRepository → SqlMemberRepository`. ViewModels kennen die SQL-Implementierung nie direkt.

`SqlRepositoryBase.WithConnectionAsync()` kapselt Connection-Open/Close zentral — kein Boilerplate in jeder Methode.

`IViewService`: Dialog-Öffnung aus ViewModels herausgehalten. Factory-Delegate-Pattern: `ViewService` bekommt `Func<AppUser, UserDetailViewModel>` — Repository-Kenntnis bleibt im Composition Root (`App.xaml.cs`).

---

### ⚠ DB-Projekte (.sqlproj) — geplant

Schema-Drift bereits aufgetreten — DB-Report veraltet. SSDT-Projekt geplant.

**Details:**  
RBAC-Tabellen (`d_role`, `d_perm`, `x_ur`, `x_rp`) sind im Code implementiert, fehlen aber im DB-Report vom 2026-03-10 — konkretes Argument für ein DB-Versionierungsprojekt.

Geplant: SSDT-Projekt anlegen — ermöglicht Schema Compare, automatisierten Deploy via `sqlpackage` und Git-Versionierung des Schemas.

---

### ⚠ Key-Index / Query-Optimierung — geplant

Häufig genutzte WHERE-Spalten ohne Index. Aktuell unkritisch — geplant.

**Details:**  
Verifiziert im DB-Report: Nur PK-Indizes (CLUSTERED auf `id`) und UNIQUE auf `gl_nr` / `win_user` vorhanden.

Fehlend: Index auf `m_gl.sta_id` (Soft-Delete-Filter in jedem `GetAllAsync`) und `f_soll.erledigt` (offene Forderungen). Bei aktuell 6/8 Zeilen irrelevant — bei wachsender Datenmenge: Full Table Scan.

### Details: Mitgliederverwaltung (CRUD)
`DeleteAsync` setzt `sta_id=6` (Ausgetreten) + `austritt=heute` — kein physisches DELETE (DSGVO). `GetAllAsync` filtert `WHERE sta_id <> 6`. Edit-Copy: Beim Bearbeiten immer Clone → auf Kopie arbeiten → bei Save AutoMapper zurück ins Original.

### Details: Gebühren-CRUD
Löschen verweigert wenn `f_ist`-Einträge vorhanden (Kassenintegrität). `CreateDefaultFeeAsync` liest `std_betrag` aus `d_ybart` — kein Hardcode. `IsDone` (erledigt) wird ausschließlich vom DB-Trigger `tr_f_ist_auto_close` gesetzt — nie per C#-Code.

### Details: Zahlungserfassung
Einzelzahlung: INSERT in `f_ist` → Trigger prüft automatisch ob Forderung gedeckt. Massenanlage: `sp_multiplier_soll` mit `BatchId (GUID)` — ermöglicht Rückgängigmachen per `DELETE WHERE batch_id = @id`.

### Details: Dashboard / KPI-Cards
Aktive Mitglieder · Offene Forderungen (Summe) · Geburtstage heute · Geburtstage nächste 7 Tage · Jubiläen (5er-Schritte). Alle 5 Queries in einer DB-Verbindung. Geburtstags-Query jahreswechsel-sicher via `DATEFROMPARTS`.

### Details: Gebührenstatus-Visualisierung
`DataTrigger` in `MembersView.xaml`: `IsOverdue=true` → orange, `IsDone=true` → grün, `IsDone=false` → rot. `IsOverdue` und `RemainingAmount` sind berechnete C#-Properties (kein eigener DB-Wert).

### Details: Benutzerverwaltung (AppUser)
CRUD auf `m_u`. Alle Schreib-Commands per `canExecute: _ => _currentUser.IsAdmin` gesperrt. Selbstlösch-Schutz: WinUser-Vergleich vor DELETE. Optional: Verknüpfung User ↔ Mitglied über `m_u.gl_id → m_gl.id`.

### Details: RBAC (Rollen + Permissions) ✓ NEU
DB: `d_role`, `d_perm`, `x_ur` (User↔Rolle), `x_rp` (Rolle↔Permission). `SqlRoleRepository`: CRUD Rollen + Permissions, `SetRolePermissionsAsync` in Transaktion. `PermissionService`: lädt User + Rollen beim App-Start, cached In-Memory — kein DB-Roundtrip bei jedem `CanExecute`-Aufruf. `CanDo(permKey)` in ViewModels mit Fallback auf `IsAdmin` (Abwärtskompatibilität). Superadmin-Rollen bypassen alle Checks.

### Details: IsAdmin-DB-Lookup ⚠ 50%
Synchroner DB-Call in `UserService.GetCurrentUser()` beim DI-Setup würde Deadlock erzeugen (UI-Thread blockiert). Workaround: Windows-Gruppen-Fallback aktiv. `PermissionService.RefreshAsync()` löst das Problem async beim App-Start.

### Kern-Tabellen mit C#-Mapping

| Tabelle | C#-Model | Zeilen | Wichtigste Spalten |
|---|---|---|---|
| `m_gl` | `Member` | 6 | `id` PK · `gl_nr` UNIQUE · `sta_id` FK · `ybart_id` FK · `eintritt` · `austritt` |
| `f_soll` | `Fee` | 8 | `gl_id` FK · `betrag` · `faellig` · `erledigt` · `batch_id` |
| `f_ist` | `Payment` | 15 | `soll_id` FK (nullable) · `betrag` · `datum` · `ykto_id` · `zart_id` |
| `m_u` | `AppUser` | 0 | `win_user` UNIQUE · `gl_id` FK (nullable) · `is_admin` |

**Details `m_gl`:** 23 Spalten total. Soft-Delete via `sta_id=6` + `austritt`. `erstellt_am` / `geaendert_am` — nicht im C#-Model, DB-seitig via `getdate()`. Computed in C#: `VollerName`, `IstAktiv`, `StatusText`.

**Details `f_soll`:** `erledigt` wird **ausschließlich** vom Trigger `tr_f_ist_auto_close` gesetzt — nie per C#! `batch_id` (GUID, nullable): gruppiert Massenanlage, ermöglicht Rückgängigmachen. `PaidAmount` ist kein DB-Feld — kommt aus `LEFT JOIN SUM(f_ist.betrag)`.

**Details `f_ist`:** `soll_id` ist nullable — freie Buchungen ohne Forderungsbezug möglich. INSERT auf `f_ist` löst Trigger aus → prüft ob `SUM(betrag) >= f_soll.betrag` → setzt `erledigt=1`. Kein CASCADE von `f_soll`: Zahlungen bleiben bei Forderungs-Löschung erhalten (Kassenprüfung).

**Details `m_u`:** `gl_id` optional — Benutzer können, müssen aber kein Mitglied sein. `is_admin`: Legacy-Flag, langfristig durch RBAC-Rollen ersetzt. `letzter_login`: nullable — wird beim Login gesetzt (noch nicht implementiert).

### Foreign Keys — CASCADE-Übersicht

| Von | Spalte | → Nach | ON DELETE |
|---|---|---|---|
| `f_soll` | `gl_id` | `m_gl.id` | CASCADE |
| `x_ur` | `u_id` | `m_u.id` | CASCADE |
| `x_rp` | `role_id` | `d_role.id` | CASCADE |
| `m_jump` | `gl_id` | `m_gl.id` | CASCADE |
| `m_mark` | `gl_id` | `m_gl.id` | CASCADE |
| `x_gm` | `gl_id` | `m_gl.id` | CASCADE |
| `x_fa` | `gl_id` | `m_gl.id` | CASCADE |
| `f_ist` | `soll_id` | `f_soll.id` | **NO ACTION ⚠** |
| `m_gl · f_soll · f_ist · m_u · d_*` | alle übrigen FKs | — | NO ACTION |

> **Warum NO ACTION auf `f_ist → f_soll`?**  
> Zahlungen (`f_ist`) sollen bei Forderungs-Löschung erhalten bleiben — für die Kassenprüfung. Das erzeugt bewusst Waisen-Einträge. Kassenprüfer muss diese separat berücksichtigen.

---

### DB-Objekte (Trigger + Stored Procedure)

| Objekt | Typ | Funktion |
|---|---|---|
| `tr_f_ist_auto_close` | Trigger (AFTER INSERT, UPDATE) | Setzt `f_soll.erledigt=1` wenn `SUM(f_ist.betrag) >= f_soll.betrag` |
| `sp_multiplier_soll` | Stored Procedure | Massenanlage `f_soll` für eine Gruppe. Parameter: `@ygrp_id, @jahr, @zweck, @batch_id` |

**Details `tr_f_ist_auto_close`:** Feuert auf `f_ist` bei INSERT und UPDATE. Prüft für die verknüpfte `f_soll`-Zeile ob die Summe aller Zahlungen den Sollbetrag erreicht oder übersteigt. Zentrale Geschäftslogik in der DB — C#-Code setzt `IsDone` niemals manuell.

**Details `sp_multiplier_soll`:** Wird aus `SqlFinanceRepository.CreateGroupFeesAsync()` aufgerufen. Die BatchId (GUID) gruppiert alle erzeugten Einträge — ermöglicht Rückgängigmachen per `DELETE WHERE batch_id = @id`.
